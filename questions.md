# Вопросы для самопроверки

1. Какие критерии вы использовали при разделении монолита на микросервисы (или при проектировании сервисов)?
**Ответ:** Разделение основано на бизнес-доменах (бронирования и доступность слотов), что позволяет масштабировать их независимо и использовать паттерн распределенной транзакции. API Gateway выделен для маршрутизации и предоставления удобного REST API клиентам, скрывая внутреннее взаимодействие по gRPC.

2. Почему вы выбрали именно этот стек технологий (язык, БД, протоколы)?
**Ответ:** Python был выбран за скорость разработки и хорошую поддержку gRPC/FastAPI. SQLite используется как легковесная база данных для каждого сервиса, чтобы показать независимость хранения данных (Database-per-service). gRPC выбран для внутреннего общения между сервисами из-за высокой производительности и строгой типизации контрактов (protobuf), а REST (FastAPI) — для внешнего интерфейса, так как он более доступен для большинства клиентов.

3. Как ваши сервисы справляются с ошибками (retry, circuit breaker, graceful degradation)?
**Ответ:** Используется паттерн компенсирующей транзакции (Saga). При ошибке сохранения бронирования в базу, вызывается отмена резервирования слота (`ReleaseSlot`), чтобы не блокировать ресурсы. Установлены таймауты на RPC вызовы.

4. Как организован деплой и обновление вашей системы без простоя (Zero Downtime)?
**Ответ:** Деплой организован через Docker Compose для локальной разработки и Helm Chart для Kubernetes. В Kubernetes это достигается за счет Deployment с настройкой Rolling Update (постепенное замещение подов) и Readiness/Liveness проб.

5. Что было самым сложным в интеграции всех компонентов вместе?
**Ответ:** Самым сложным было обеспечение согласованности данных между сервисами при ошибках и настройка корректной связи между контейнерами (Service Discovery внутри docker-compose и Kubernetes), а также обработка grpc-ошибок.

6. Что бы вы улучшили, если бы у вас был еще месяц?
**Ответ:** Так с ходу и не сажешь, ведь технологии, которые тут можно применить пока возмоно и не известны, но приоритетными задачами можно поставить улучшение frontend состовляющей и более предусмотрительное логирование ошибок.
