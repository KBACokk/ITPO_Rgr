# Описание CI пайплайна (bookings-s11)

Процесс непрерывной интеграции (CI) и непрерывной доставки (CD) организован с использованием GitHub Actions. Пайплайн состоит из двух основных этапов (jobs): `lint-and-test` и `build-and-publish`.

## Этап 1: Линтинг и Тестирование (lint-and-test)

1. **Подготовка окружения**: Код скачивается (checkout), настраивается Python 3.10.
2. **Установка зависимостей**: Устанавливаются зависимости, включая `flake8` и `pytest`.
3. **Линтинг**: Происходит проверка качества кода с помощью `flake8` (проверка синтаксиса и стиля).
4. **Тестирование**: Запускаются юнит-тесты с помощью `pytest` (`make test WEEK=17`).
5. **Артефакты**: Отчеты о тестах сохраняются как артефакты для последующего анализа.

## Этап 2: Сборка и Публикация Docker-образа (build-and-publish)

Этот этап запускается только при успешном прохождении `lint-and-test`.

1. **Docker Buildx**: Настраивается среда для сборки Docker-образов.
2. **Авторизация в Docker Hub**: С использованием секретов GitHub Actions (`DOCKERHUB_USERNAME`, `DOCKERHUB_TOKEN`) происходит вход в реестр.
3. **Сборка и Пуш (Build and Push)**: Собираются образы микросервисов (`bookings-svc`, `availability-svc`, `gateway`) с использованием кеша для ускорения (gha cache). После сборки образы пушатся в Docker Hub с тегами `latest` и тегом, равным хэшу коммита.

Это позволяет гарантировать, что в `main` ветку попадает только протестированный код, который автоматически собирается и публикуется для дальнейшего развертывания в Kubernetes.
